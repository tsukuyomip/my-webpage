# 学マス サポカ棚卸し (gkms-sprtcrd)

学園アイドルマスターの「所持サポートカード一覧」スクリーンショットを画像認識し、
[学マスwiki（Seesaa）のサポートカード一覧](https://seesaawiki.jp/gakumasu/d/%A5%B5%A5%DD%A1%BC%A5%C8%A5%AB%A1%BC%A5%C9%B0%EC%CD%F7)
を母集団として、**カード名（wiki 表記のまま）・Lv・凸数・タイプ・上限解放可能** を
一覧化する Web アプリ。GitHub Pages 上で完全にクライアントサイドで動く。

## 使い方

1. **① wikiから取得** — カード一覧ページを CORS プロキシ経由で取得してパース
2. **② カード画像を取得** — 各カードの画像を取得し、照合用シグネチャ
   （dHash + 色グリッド）を計算。一覧表の画像は取得できないことがあるため、
   **カード詳細ページを 1 階層クロールして固定リンク**
   （`image0N.seesaawiki.jp/g/u/gakumasu/<hash>.png`）を解決して使う。
   解決した画像 URL・シグネチャは IndexedDB に永続キャッシュされ、
   2 回目以降は新カードの分しか wiki にアクセスしない。失敗したカードは
   理由付きで画面に一覧表示される
3. **③ スクショをドロップ**（複数可・Picsew 等の縦結合スクショ可）
4. **④ 結果テーブル** で確認。低信頼のマッチや読めなかった Lv は ⚠ 付きで出るので、
   プルダウン（マスタ全カードから選択）や直接入力で修正
5. クリップボード（TSV）/ CSV / JSON でエクスポート

## アーキテクチャ

```
src/lib/
  geometry.ts     グリッド・オーバーレイの座標系（実機スクショ計測値、画像幅比）
  screenshot.ts   解析コア（グリッド検出・凸・タイプ・上限解放・OCR前処理）
                  ※ DOM 非依存の純粋関数群。vitest + pngjs で実データ回帰テスト
  ocr.ts          Lv 数字の OCR（tesseract.js、アセット自前配信）
  hash.ts         dHash + 4x4 色グリッドのシグネチャと距離
  match.ts        照合（タイプ整合ペナルティ、ベスト/次点マージンで信頼度）
  wikiParser.ts   Seesaa wiki HTML → カードマスタ（防御的パース）
  proxy.ts        CORS プロキシ層（複数候補 + カスタム設定、EUC-JP デコード）
  cache.ts        IndexedDB（マスタ TTL 24h、画像シグネチャは永続）
  masterSource.ts マスタ取得の抽象化（live / HTML インポート / baked）
  analyze.ts      スクショ → 解析結果のパイプライン（ブラウザ糊）
```

### 解析の仕組み（スクショ側）

- **グリッド検出**: サムネイル右下のタイプアイコン（色付き円盤）を各列の
  固定 x で縦に全走査し、「内側が単一タイプ色・外側は違う色」の円盤チェックが
  通る y をクラスタ化して行位置を求める。行ピッチに依存しないので、
  結合スクショの継ぎ目でピッチが崩れていても動く。
  さらに、アイコン中心からセル上端へは約 0.8 セル分「上向きに外挿」するため
  幅由来の高さ誤差が上端で増幅されがち。**カード上辺（背景→絵柄の遷移）を検出して
  上端をスナップし、高さをアイコン位置から逆算**する自己補正を入れ、行間の余白へ
  食い込む切り取りズレを防ぐ
- **凸数**: 解放済みアイコンは「帯状に途切れない白の縦ラン」で判定。
  凸アイコン列は Lv の桁数で約 30px 左右にずれるため、先に OCR した Lv の
  桁数でグリッドを確定する（OCR 失敗時はヒューリスティックにフォールバック）
- **Lv OCR**: 数字領域を 2 値化 → 背の高い連結成分だけ残す（帯の白文字・
  凸アイコン断片・アートの白を除去）→ 3 倍拡大 + 余白 → tesseract.js
  （数字ホワイトリスト）
- **レアリティ帯**: サムネ最下端の帯色から R/SR/SSR を判定（**R=水色 / SR=金 /
  SSR=虹**）。虹は色相のばらつきが大きい・金は金色相に集中・水色は低彩度シアン、
  という特徴で分類（フィクスチャで回帰テスト済み）
- **照合**: セルのアート上部（オーバーレイの無い領域）と wiki 画像の同一相対領域を
  dHash + 色グリッドで比較。**照合領域は「上半分かつ右上隅を除く」**を基本とし、
  高レベル/高凸/上限解放帯で埋まる下半分に加え、右上に乗ることのある
  未読バッジ（赤丸）も距離計算から静的マスクで除外する（マスタ・スクショ双方に
  対称にかかるので既存 baked 署名の作り直しは不要）。
  さらに **「上限解放可能」帯が無いカードは上 2/3（中段まで）を使って照合**する。
  マスタ側の上 2/3 署名（`signature.tall`）は起動時に同梱画像から計算し IndexedDB に
  キャッシュ（全カードに揃った時だけ tall 経路を使い、距離スケールの整合を保つ）。検出タイプ・レアリティが
  wiki と食い違う候補にはペナルティ（**レアリティ帯は色分離が明瞭で信頼度が高い
  ため、上半分がそっくりな別レアリティのカードを弾ける**）。ベストと次点の距離差が
  小さい場合は low confidence として警告。手動修正のプルダウンも推定レアリティを
  先頭に並べる

## wiki への配慮と「腹持ち（ベイク）方式」への移行メモ

現状は**閲覧時にプロキシ経由で取得**する方式だが、以下で負荷を抑えている:

- ページ HTML はパース結果を IndexedDB にキャッシュ（24h 経過で再取得を促すだけで、
  自動再取得はしない）
- カード画像はシグネチャ（数十バイト/枚）だけ永続保存し、画像そのものは保持しない。
  一度計算したカードには二度とアクセスしない
- 詳細ページのクロール結果（解決した固定画像 URL）も card 単位で永続キャッシュ。
  詳細ページの取得も一度きり
- 画像・詳細ページの取得は同時 1 本 + 400ms 間隔に制限

> ⚠ **重要（Seesaa の IP ブロック）**: seesaawiki.jp は
> データセンター/クラウドの IP（**GitHub Actions・Cloudflare 等**）からの
> アクセスを **403 Forbidden** で拒否する。そのため下記 A0（Actions 自動）や
> サーバレス Worker 経由の自動取得は基本的に通らない。
> **確実なのは「手元PC（住宅回線）から実行する下記 A」**。実測で、住宅回線は
> 200 で 192 カード取得、Actions ランナーは 403（本体HTMLが返らない）だった。

**基本セット（腹持ち）の更新 A0 — GitHub Actions（※Seesaa に 403 されがち）:**
`.github/workflows/bake-gkms.yml` が `scrape-wiki.mjs` をランナー上で実行し、
変更があれば自動コミット → Pages デプロイを起動する。仕組みとしては動くが、
**Seesaa がランナー IP を 403 する場合は取得できない**（その旨をログに明示する）。
将来 Seesaa の挙動が変わった時や、セルフホストランナー（住宅回線）を使う場合の
受け皿として残してある。

- 実行方法（②手動）: **Actions タブ → "Bake gkms-sprtcrd master data" → Run workflow**
- 定期自動（①）へは `schedule:` の2行のコメントを外すだけ
- ページ側にトークンは置かない（Actions 内 `GITHUB_TOKEN` のみ）

**基本セット（腹持ち）の作り方 A — スクレイパを手元で実行（最推奨・確実）:**
`scripts/scrape-wiki.mjs` は実ブラウザ（Playwright）で一覧ページを開き、
**表のサムネイル画像だけ**を取得して baked セット
（`public/baked-master.json` + `public/card-images/`）を直接生成する。
CORS プロキシを介さず直接アクセスするので、アプリ内②のプロキシ失敗を回避できる。
署名計算はアプリと同一アルゴリズムをブラウザ内で実行するため、生成物は
そのまま照合に使える（実データ検証で screenshot 照合 53/53 一致を確認）。

```bash
cd gkms-sprtcrd
npm install
npm install -D playwright          # このスクリプト用（一度だけ）
npx playwright install chromium    # 一度だけ
node scripts/scrape-wiki.mjs --limit 10   # まず動作確認
node scripts/scrape-wiki.mjs              # 全カード → public/ に出力
# public/baked-master.json と public/card-images/ をコミット
```

※ Playwright はビルド/デプロイを重くしないため通常依存には入れていない
（使うときだけ `npm install -D playwright`）。この一覧取得は詳細ページを
クロールせず、表のサムネイルのみを取得する。

**基本セット（腹持ち）の作り方 B — アプリの画面から:**
開発環境（サンドボックス）からは seesaawiki.jp の egress がブロックされているため、
crawl/bake はできない。代わりに**ブラウザで一度 ②を完走**してから、診断パネルの
いずれかで書き出してコミットする:

- **画像込みZIPをエクスポート（推奨）**: `baked-master.json` + 実画像
  （`card-images/*`）を 1 つの zip に固めて出力。zip の中身を
  `gkms-sprtcrd/public/` に展開してコミットすると、画像も自オリジン配信の
  完全自己完結セットになる（wiki にもプロキシにも一切触れない）。
  baked JSON 内の画像 URL はローカルパスに書き換えられる。
- **マスタJSONエクスポート**: シグネチャのみの軽量版。照合はこれだけで動く
  （画像はリポジトリに置かない）。

どちらも起動時に `public/baked-master.json` が最優先で読み込まれ、差分の
新カードだけ ②で追加取得できる。zip は依存なしの自前 store zip
（`src/lib/zip.ts`）で生成している。

**将来、事前に焼き込んだデータ（腹持ち）に切り替える手順:**

1. アプリの「診断/上級者向け → マスタJSONエクスポート」で
   `baked-master.json`（マスタ + 全シグネチャ）をダウンロード
2. それを `gkms-sprtcrd/public/baked-master.json` としてコミット
3. 以後、起動時に同梱データが最優先で読み込まれ、wiki アクセスは
   「wikiから再取得」を手で押した時だけになる

CI（GitHub Actions の cron）で定期ベイクする場合も、同じ JSON
（`masterSource.ts` の `BakedMaster` 形式 `gkms-sprtcrd-baked-master@1`）を
生成して同じ場所に置けばよい。取得層は `masterSource.ts` に集約してあり、
live / imported-html / baked のどれでも同じ `MasterData` に正規化される。

## 既知の制約・未検証事項

- **開発環境から seesaawiki.jp に到達できなかった**（サンドボックスの egress 制限
  + wiki 側の bot 拒否）ため、`wikiParser.ts` は Seesaa wiki の一般的な構造
  （テーブル内に画像 + カード名リンク）を仮定した防御的実装になっている。
  **初回の実利用でパース結果（診断パネルの件数・先頭20件）を必ず確認すること。**
  想定とズレていた場合は、ブラウザで保存した HTML を診断パネルからインポートすれば
  パーサのデバッグ用データになる
- 公開 CORS プロキシ（allorigins / corsproxy.io）が wiki 側にブロックされる
  可能性がある。その場合は「保存HTMLをインポート」フォールバック +
  カスタムプロキシ設定（自前 Cloudflare Worker 等）で回避
- wiki のカード画像がゲーム内サムネイルと同じ構図（横長フルアート）である前提。
  枠付き画像などだった場合は照合精度が落ちる（診断は結果テーブルの信頼度で可能）
- Lv OCR の精度は実測でおおむね 9 割強。読めない/範囲外のものは ⚠ を付けて
  手動修正に回す設計
- タイプアイコンの色相分類のうち、実データで検証済みなのはボーカル（ピンク）のみ。
  ダンス（青）・ビジュアル（黄）・アシスト（緑）は色相範囲から推定
  （サンプルスクショがボーカル絞り込み表示だったため）
